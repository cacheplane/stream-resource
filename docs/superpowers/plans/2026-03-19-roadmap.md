## Part 2 — Inferred Next Steps

These are logical continuations of the project's direction. They're not explicitly requested but follow naturally from what has been built and what a commercial library launch requires.

---

### 2.1 Lead Form — Email Delivery

**Why:** The `/api/leads` route currently logs to `console.info` and returns 200. In production, leads are silently dropped — nothing reaches Brian's inbox.

**What to do:**
- Integrate a transactional email service (Resend is the simplest option for Next.js; alternatively Postmark, SendGrid)
- On POST to `/api/leads`, send an email to `hello@cacheplane.ai` with the lead details
- Optionally: also add to a CRM or Airtable for tracking
- Add `RESEND_API_KEY` (or equivalent) to Vercel environment variables

**Effort:** Small (1 session)

---

### 2.2 Payment Flow for Commercial Licenses

**Why:** The pricing page has "Buy License" CTAs that link to `#lead-form`. There's no way to actually purchase a Developer Seat ($500/seat/year) or App Deployment ($2,000) without going through a manual email flow. For a commercial product at these price points, that's fine initially — but at scale it becomes a bottleneck.

**Options:**
- **Lemon Squeezy** — purpose-built for software licenses, handles VAT, generates license keys, works well for seat-based products
- **Stripe** — more flexible, requires more integration work
- **Manual initially** — invoice via Stripe Billing on request, upgrade later

**What to do (manual-first approach):**
- Keep the lead form as the purchase trigger
- Create Stripe products for each tier for invoice generation
- Revisit full checkout automation at $X MRR threshold

**Effort:** Medium

---

### 2.3 Versioning Strategy and Release Process

**Why:** The library is at `0.0.1` with no changelog, no release notes, and a `publish.yml` that runs on any `v*` tag push. A library used in enterprise production needs a predictable release process.

**What to build:**
- `CHANGELOG.md` — conventional-commit based, maintained per release
- Semantic versioning policy: what constitutes a major/minor/patch bump for this library
- Tag-based release process: `npm version patch` → `git push --tags` → CI publishes
- GitHub Releases (auto-generated from tag + changelog section)
- Consider Nx Release (`nx release`) for coordinated multi-package versioning

**Effort:** Small–Medium

---

### 2.4 Library v1: Complete the Known Gaps

**Why:** `docs/limitations.md` documents three v1 gaps that return empty unconditionally. These are the most significant parity gaps with React `useStream()`.

**Items:**

| Gap | What's needed |
|---|---|
| `toolProgress()` always returns `[]` | Implement `tool_progress` event handling in `stream-manager.bridge.ts` |
| `toolCalls()` always returns `[]` | Implement `tool_calls` event handling |
| `getMessagesMetadata()` always returns `undefined` | Implement `onMessagesMetadata` callback integration |
| `queue` not exposed | Expose queue inspection signal (lower priority) |

**Effort:** Medium (requires careful SDK type verification)

---
